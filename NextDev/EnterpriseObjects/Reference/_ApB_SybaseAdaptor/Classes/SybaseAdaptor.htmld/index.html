SybaseAdaptor



Inherits From: EOAdaptor : NSObject
Declared In: eoadaptors/Sybase/SybaseAdaptor.h



Class Description

A SybaseAdaptor represents a single connection to a Sybase database server, and is responsible for keeping login and model information, performing Sybase-specific formatting of SQL expressions, and reporting errors.

The features SybaseAdaptor adds to EOAdaptor are as follows:

Sybase-specific message and error handling
The ability to configure the sharing of DBProcesses
The ability to configure the maximum textsize
The ability to specify a client character set and language
Sybase password encryption

The SybaseAdaptor class has these restrictions: A context can only manage one channel at a time, and the adaptor doesn't support full outer joins because the Sybase server itself doesn't support them.

The Connection Dictionary

The connection dictionary contains items needed to connect to a Sybase server, such as the server name and database (it's common to omit the user name and password from the connection dictionary, and prompt users to enter those values in a login panel). The keys of this dictionary identify the information the server expects, and the values of those keys are the values that the adaptor uses when trying to connect to the server. For Sybase databases the required keys are as follows:

hostName
databaseName
userName
password

The connection dictionary can optionally include three other keys: sybasePasswordEncryption, sybaseClientCharacterSet, and sybaseClientLanguage. sybasePasswordEncryption provides support for Sybase password encryption. sybaseClientCharacterSet declares to the Sybase server the character set being used by the client (such as eucjis, ascii7, or iso_1), while sybaseClientLanguage specifies the language in which you would like server error messages to appear. See your Sybase documentation for a complete list of types available for these two fields.

To add these keys and appropriate values to your connection dictionary, you must manually edit your model file. For example:

connectionDictionary = {databaseName = People;
hostName = "";
password = "";
userName = sa;
sybaseClientCharacterSet = eucjis;
sybaseClientLanguage = us_english;
sybasePasswordEncryption = YES;
};

Subsequently changing the connection dictionary in your model file using the Set Adaptor Info command in EOModeler has no effect on these keys and their values--they are preserved unless you edit the file to remove them.

The default character set for non-Japanese systems is iso_1 (that is, ISO Latin 1), while the default character set for Japanese systems is eucjis. You only need to add the sybaseClientCharacterSet key to your connection dictionary if you are using a character set other than your system's default.

Note:  Enterprise Objects Framework uses NEXTSTEP encoding to represent string data, and it passes strings to the database without converting them to the database character set. If you require that the data passed to your server is in an encoding other than NEXTSTEP encoding, you need to subclass NSString.



Error and Message Handling

The Sybase adaptor and individual Sybase channels can be configured to handle errors and messages from the Sybase server in different ways. When an error or message is received from the server on a DBProcess associated with a SybaseChannel, the channel is consulted for the appropriate behavior. When no channel is present, the SybaseAdaptor class is consulted. The setLogsErrors: and setLogsMessages: methods allow you to specify how errors and messages are handled. When you create channels, they're initialized with the values of their parent adaptor. Changing an adaptor's settings affect channels created after the change, but previously created channels aren't affected.

The SybaseChannel class provides delegate methods for handling messages and errors, and for processing compute rows and stored procedures. For more information, see the SybaseChannel class specification.



Using DBProcesses

You can configure a SybaseAdaptor object to share DBProcess data structures among contexts. Sharing DBProcesses is useful if you have created several channels but only a few are actually in use at one time, or if you destroy and create multiple channels. When an adaptor is configured to share DBProcesses, channels ask the adaptor for a DBProcess structure before each transaction, and give the DBProcess back to the adaptor after each transaction has completed. You use setDBProcessPoolLimit: to specify the maximum size of the pool of DBProcesses the adaptor maintains.

When an adaptor is configured to share DBProcesses, you should never do anything that sets the state of a DBProcess beyond its current transaction (for example, by sending the statement "set textsize 10" with a channel's evaluateExpression: method). This is because the DBProcess may be given to another context in a subsequent transaction.

DBProcess structures are an expensive resource in the Sybase server, and opening and closing them are expensive operations. You should carefully weigh the trade-offs between opening and closing a DBProcess multiple times against the expense of keeping several of these resources open for the life of your application.



Locking

All adaptors use the database server's native locking facilities to lock rows on the server. The Sybase adaptor locks a row by using the HOLDLOCK keyword in SELECT statements. This occurs when:

You send the adaptor channel a selectAttributes:describedByQualifier:fetchOrder:lock: message with YES specified as the value for the lock: keyword.
You explicitly lock an object's row with the lockObject: message.
You set pessimistic locking at the database level and fetch objects.

The semantics of the HOLDLOCK keyword are such that when you lock a row other users can't update it, but it doesn't guarantee that your update will succeed. This is because other users could be holding a lock on the same row. However, you can still read rows that are locked by other users.



SQL and User-Defined Transactions

Certain data definition commands, such as CREATE TABLE, can't be executed in a user-defined transaction. However, the database channel and adaptor channel require you to start a transaction before evaluating any SQL. To work around this problem, you need to send the adaptor context or database context a transactionDidBegin message to make it think a transaction is in progress. Then you can send it the SQL statement, followed by a transactionDidCommit message.

Returns the number of seconds the client library waits for a login response before timing out. A return value of 0 indicates no timeout limit.



logsErrors
+ (BOOL)logsErrors

Returns YES if the adaptor logs errors, NO if not. Errors are logged to the console. The default is YES.



logsMessages
+ (BOOL)logsMessages

Returns YES if the adaptor logs messages, NO if not. Messages are logged to the console. The default is YES.

Instance Methods

dbProcessPoolLimit
(int)dbProcessPoolLimit

Returns the maximum number of DBProcesses that can be in the DBProcess pool.



maxTextSizeDefault
(int)maxTextSizeDefault

Returns the maximum number of bytes that can be returned from a Sybase image-to-text field. The default is set to INT_MAX as defined for the host machine. This number can be overwritten on a channel-by-channel basis by sending the appropriate SQL to the channel using the evaluateExpression: method.



setDBProcessPoolLimit:
(void)setDBProcessPoolLimit:(int)limit

Sets to limit the maximum number of DBProcesses that can be in the DBProcess pool. By default, the maximum number is 0. If you specify a value of YES for setShareDBProcesses:, you must use this method to specify a positive pool limit.



setMaxTextSizeDefault:
(void)setMaxTextSizeDefault:(int)textSize

Sets the default textsize to textSize.



setShareDBProcesses:
(void)setShareDBProcesses:(BOOL)flag

Sets according to flag whether the adaptor shares DBProcesses or not. This method must be invoked before any SybaseContexts are instantiated. If you configure the adaptor to share DBProcesses, you must use setDBProcessPoolLimit: to supply a positive value for the pool size.



shareDBProcesses
(BOOL)shareDBProcesses

Returns YES if the adaptor has been configured to share Sybase DBProcesses, NO if not. The default is NO.



typeForName:
(int)typeForName:(NSString *)typeName

Returns an integer constant representing the external type typeName.
